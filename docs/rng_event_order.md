# RNG Event Order (Implementation Notes)

This document describes how Pylatro RNG event pulls are ordered to mirror Balatro's seeded behavior.

## 1. Core stream model

Balatro uses named RNG streams. Every pull is keyed and advances that key's internal state.

- `pseudoseed(key)` initializes stream state from `pseudohash(key .. seed)` (lazy first use).
- Every call advances the key state with:
  - `(2.134453429141 + state * 1.72431234) % 1`
  - rounded through `string.format("%.13f", ...)`
- Output is blended with `hashed_seed` and then used to seed `math.random`.

Call order matters. If you consume one key earlier/later, all subsequent values for that key diverge.

## 2. Startup event order

At run initialization, the RNG order must be:

1. Boss selection (`boss` stream)
2. Voucher selection (`Voucher` stream)
3. Small blind tag (`Tag` stream)
4. Big blind tag (`Tag` stream again)

Pylatro helper methods in this order:

- `startup_boss()`
- `startup_voucher()`
- `startup_small_tag()`
- `startup_big_tag()`

## 3. Joker and consumable pull order

For generated shop cards:

1. Determine rarity with key `rarity{ante}{append}`
2. Select pool key `Joker{rarity}{append}{ante}`
3. Build candidate list where unavailable entries become `UNAVAILABLE`
4. Pull from candidate list with `pseudorandom_element`
5. If selected value is `UNAVAILABLE`, retry with `_resampleN` suffix:
   - `{pool_key}_resample2`, `{pool_key}_resample3`, ...
6. After card type decision, poll edition with key `edi{append}{ante}`

Consumables follow the same pull pattern using their type pool key:

- Tarot: `Tarot{append}{ante}`
- Planet: `Planet{append}{ante}`
- Spectral: `Spectral{append}{ante}`

## 4. Booster pack pull order

Booster pack selection uses weighted polling:

1. Compute total weight of eligible packs
2. Pull `poll = random(pack_generic{ante}{append}) * total_weight`
3. Traverse cumulative weight in deterministic order
4. Select first pack crossing the threshold

## 5. Erratic deck order

Erratic deck faces are generated by repeatedly pulling from the playing card pool using the same key:

- Key: `erratic`
- Repeated 52 times for a starting deck
- Each pull advances the `erratic` stream

## 6. Random event keys (reactive gameplay)

Current implementation includes helpers for these event streams:

- `lucky_mult`
- `lucky_money`
- `glass`
- `wheel_of_fortune`

Blind targeting and other stream keys can be added with the same pattern.

## 7. Determinism checklist

To keep reproducibility:

- Keep pool ordering stable and explicit.
- Never rely on unordered dict iteration for pool traversal.
- Preserve exact stream call order per game event.
- Snapshot/restore full RNG state when saving/loading runs.

## 8. Content-backed pool verification

RNG pools are loaded from Pylatro content files/repository data:

- jokers: `content/data/jokers.txt` (grouped by rarity)
- vouchers: `content/data/vouchers.txt`
- tags: `content/data/tags.txt` (min ante filtering applied)
- tarots/planets/spectrals: content lists from corresponding files
- bosses: `content/data/boss_blinds.txt`

This means report lines are directly verifiable against the project data source.

## 9. Non-erratic deck pull verification

The seed report includes a non-erratic deck section:

- `[standard_deck:<name>]`
- `draw=<card1,card2,...>`

Cards are taken from deck combination files and shuffled with key `shuffle`:

- `content/data/deck_combinations/default.txt`
- `content/data/deck_combinations/abandoned.txt`
- `content/data/deck_combinations/checkered.txt`
